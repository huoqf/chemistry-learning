# 勒夏特列原理重构问题深层次复盘与预防策略报告

本次“勒夏特列原理”重构过程中暴露了多个涵盖 UI 样式、动画数据流、物理化学正确性以及侧屏排版的典型缺陷。为杜绝此类问题再次发生，并确保项目规范与 Skill 文件不发生膨胀与冲突，特作以下深度复盘。

---

## 一、 问题根因归因分析 (Root Cause Analysis)

| 问题现象 | 表面原因 | 根本原因 (Root Cause) |
| :--- | :--- | :--- |
| **1. 初始深色背景** | 代码手写了 `bg-slate-900/40` | **规范遵从度缺失**：未强制继承系统的 Light Theme 规范，组件层自行拼装非标样式。 |
| **2. 工业喷头遮挡文字** | 集气瓶中央挂着 `ParticleEmitter` | **语义匹配度审查缺失**：引入器材前未核对语义。`ParticleEmitter` 是带机械外壳的工业发射枪，误用于盛放气体的简易集气瓶。 |
| **3. 图表拉斜/拉到一起** | 动态采样 `0..time` + 自动定标 | **图表定标与揭示未解耦**：计算 Hook 将采样范围与当前 `time` 绑定；同时图表未锁定 `xDomain={[0, 10]}`，导致 X 轴随时间拉伸。 |
| **4. 5 秒强行中断** | 注册项配置了 `stopCondition: t>=5.0` | **模式选用错误**：把持续演化型过程动画当作了“中途离散停止”过程。 |
| **5. 平衡提示消不掉** | 提示出现在底栏且重置未清空 | **状态生命周期遗漏**：`handleReset` 与 `setParams` 动作未清空 `equilibriumReached` 状态。 |
| **6. 侧屏横向滚动条** | $Q_c$ 关系式在一行中用超长 `\text{}` | **LaTeX 文本滥用**：滥用 `\text{}` 包裹长段中文，KaTeX 无法自动换行导致挤爆右屏宽度。 |
| **7. 放热/吸热方向颠倒** | 正向写成了“吸热”，逆向写成了“放热” | **化学逻辑核对缺失**：代码编写后未对照真实化学反应方程式（$2\text{NO}_2 \rightleftharpoons \text{N}_2\text{O}_4, \Delta H < 0$）逐字校验。 |

---

## 二、 以后如何避免类似问题？(Preventive Actions)

针对上述根因，后续开发与重构过程必须落实以下 5 项防御机制：

### 1. 严格执行“组件语义与视觉审查”
- **禁止盲目拼装器材**：在 `src/components/Chemistry` 中选用器材时，必须核对组件的实际视觉样式。简易集气瓶内部的气体/溶液，统一使用简单 SVG 粒子点阵或半透明 fill，严禁混入自带重型外框的工业机械组件。
- **UI 纯洁性**：动画画布外框必须统一使用系统标准 `ThreePanel` 或 `AnimationSvgCanvas` 的通用浅色样式（Light Theme），禁止任何组件自行写入 `bg-slate-900` 等深色背景。

### 2. 贯彻“图表坐标轴定标与揭示解耦”铁律
- **固定物理时间轴**：所有时序演化图表（$v-t$, $c-t$, $p-t$ 等），计算 Hook 返回的 `history` 必须基于固定的全量物理时间轴（如 $0 \sim 10\text{s}$），严禁根据 `time` 动态改变采样时间区间！
- **显式锁定 `xDomain`**：图表组件必须显式传入 `xDomain={[0, maxTime]}`。随 `time` 推进时，仅使用 `.filter(p => p.time <= time)` 动态显现已走过的轨迹，确保 X 轴刻度恒定不动，曲线如仪器记录般平滑生长。

### 3. 规范提示生命周期与归属
- **底栏纯粹化**：`AnimationControls` 仅负责播放/暂停/重置/倍速，不得承载具体的化学状态提示。
- **主屏 SVG 标示**：所有针对场景状态的提示（如“已达新化学平衡”），统一在主屏 SVG 画布上方以微型 Badge 形式展示。
- **状态原子重置**：所有重置动作（`handleReset`、`setParams`、`updateParam`）必须连同 `equilibriumReached` 状态一并清空。

### 4. 严禁 LaTeX 中文文本滥用与侧屏溢出
- **侧屏无滚动条原则**：侧屏宽度有限，所有公式必须经过宽度校验。
- **禁止段落型 LaTeX**：严禁在 `latex` 属性中使用 `\text{...}` 包裹超过 6 个汉字的大段段落。多条件判据必须**拆分为多行独立的简短公式**，保证在任何分辨率下均不产生横向滚动条。

### 5. 提交前“化学逻辑逐字对照”
- 涉及热化学（$\Delta H$）、勒夏特列移动、酸碱滴定、电极极性等反应时，开发完成后必须对照高中化学教材进行“反应方向 - 吸/放热 - 分子数变化”的逐字三向核对。

## 三、 关于项目规范与 SKILL 修改的原则 (Skill Governance)

针对您提出的**“改项目规范和 SKILL 务必小心，仔细综合分析，严禁随意局部修改，以免出现冲突、重复，文件膨胀”**：

我们深表认同！盲目追加规则会导致 SKILL 极其臃肿、逻辑碎片化、甚至前后矛盾。

在后续沉淀或更新 SKILL / AGENTS 规范时，我们将严格遵循以下治理原则：

1. **归纳提炼而非零散罗列**：
   不为单个具体特例追加规则（如“不要把发射枪放在集气瓶里”），而是归纳为抽象且有约束力的机制规范（如“组件语义核对规范”与“图表定标与揭示解耦规范”）。
2. **结构化收聚与去重**：
   修改 SKILL.md 时，先审查现有章节结构（Step 0 审计、Step 1 布局、Step 3 图表等），将新提炼的防错机制归缩在现有对应章节内，**绝不新增零碎的临时章节**。
3. **保持全局单一事实来源 (Single Source of Truth)**：
   通用的主题与配色规范统一收录于 `AGENTS.md`，具体的动画重构步骤统一收录于 `refactor-animation-page/SKILL.md`，禁止在多个文件间复制粘贴相同的文字，防止未来出现修改不同步导致的规则冲突。
